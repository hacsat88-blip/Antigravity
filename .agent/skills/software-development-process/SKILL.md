# フェーズ0：要件確認（Discovery）

**目的**：作るもの・作らないものを決め、事故要因を先に潰す

## 成果物（DoD）

- ゴール（何ができれば完了か）
- スコープ（対象/対象外）
- 制約（期限、予算、技術、運用）
- 受け入れ条件（テスト観点でも可）
- 主要リスク（最低3つ）と回避策

## 進め方（最小手順）

1) 参照のみで現状把握（構造、既存仕様、制約）
2) 不明点を列挙し、確認が必要なら質問を作る
3) DoD を埋め、合意できる形で短く提示
4) 状態変更が必要になったら、Rules の「承認提示テンプレート」で y/n を取ってから進める

# フェーズ1：設計（Design）

**目的**：実装を迷わない状態にする

## 成果物（DoD）

- アーキテクチャ概要（依存関係、境界、データ流れ）
- 主要API/IF（入力・出力・失敗時）
- セキュリティ/権限/データ保持の方針（必要なら）
- 運用観点（ログ、監視の最小セット）

## 進め方（最小手順）

1) 既存構成と制約を前提に、最小の設計案を作る
2) 代替案がある場合は 2〜3案とトレードオフを併記
3) DoD を満たした設計メモを提示
4) 実装に進む場合、状態変更が必要なら承認（y/n）を取る

# フェーズ2：実装（Build）

**目的**：最短で動く形を作り、継続的に品質を担保

## 成果物（DoD）

- 実装（最小動作 + 例外処理）
- 単体テスト（重要ロジック中心）
- Lint/型チェック（可能な範囲で）
- 変更点の要約（何が変わったか）

## 進め方（最小手順）

1) 変更範囲を小さく切り、差分が読みやすい粒度で進める
2) 状態変更（編集・実行）が必要なら、実行計画を提示して y/n を取る
3) DoD を満たすまで反復（テスト・Lint 等）
4) 変更点の要約を日本語で提示

# フェーズ3：検証（QA）

**目的**：本番投入に耐える最低ラインを満たす

## 成果物（DoD）

- 再現可能なテスト手順（手動/自動）
- 重要な非機能（性能/セキュリティ）の確認結果（必要な範囲）
- 既知の制約・未対応（残課題）リスト

## 進め方（最小手順）

1) 受け入れ条件に紐づくテスト観点を整理
2) 重大リスク（破壊的変更・権限・データ損失）を重点確認
3) 結果を DoD 形式でまとめ、残課題を明示

# フェーズ4：リリース/運用（Release & Ops）

**目的**：安全に出し、戻せるようにする

## 成果物（DoD）

- デプロイ手順（実行コマンド含む）
- ロールバック手順
- 監視（最低限のログ/メトリクス/アラート方針）
- Runbook（よくある障害の対応手順）

## 進め方（最小手順）

1) デプロイとロールバックを「コピペ可能な手順」に落とす
2) 監視・ログの最小セットを定義（何が壊れたら検知するか）
3) Runbook を最小で用意（一次対応→切り戻しまで）
4) 実運用コマンド実行は必ず事前承認（y/n）
