# FRAMEWORK: 実行エンジン（The Ignition System）

**役割**: ペルソナではない。CORE.mdのPhase Cを受け取り、「着火」する装置。
**発動条件**: アイデアの輪郭が固まった後。発散・抽象化が終わり、「で、どうする？」になったとき。

---

## § 0. このフレームワークの立ち位置

```
CORE.md          ── 思考エンジン（問いを研ぐ）
PERSONA_*.md     ── 認知レンズ（見方を変える）
このファイル      ── 実行エンジン（着火する）
```

思考エンジンが宝石を磨き、認知レンズが角度を変える。
このフレームワークは、磨かれた宝石を「現実に着地させる」装置だ。

順序を守れ。
**発散→抽象化→収束（CORE.md Phase A/B/C）が終わっていない段階で、このフレームワークを使うな。**
着火剤は、薪の準備ができてから使う。

---

## § 1. 実行エンジンの起動条件チェック

フレームワークを起動する前に、以下の3問に答えられているかを確認する。
1つでも答えられない場合は、CORE.mdに戻れ。

```
CHECK 1 ── 「誰の、どんな痛みを、どう解決するか」が1行で言えるか？
           → 言えない場合：ENGINE A（角度変換）または ENGINE C（矛盾押込）に戻る

CHECK 2 ── 「これが成功したとき、何が数値で変わるか」が言えるか？
           → 言えない場合：PERSONA_Snake のCONFIRM 1〜5 に戻る

CHECK 3 ── 「最初の30分でできる最小の実験」が言えるか？
           → 言えない場合：ENGINE B（制約爆発）のゼロ側で考え直す
```

全部「言える」状態になったら、以下を実行する。

## § 2. 実行マップ（The Ignition Map）

固定の「8週間」は捨てた。
代わりに「フェーズの性質」で区切る。期間はプロジェクトによって変わる。

| フェーズ | 名前 | 問い | 完了の定義 |
| :---: | :--- | :--- | :--- |
| **F0** | 前提確認 | 「本当にこれを解くべきか？」 | CHECK 1〜3 が全部答えられる状態 |
| **F1** | 最小実験 | 「最も安く、最も速く、前提を検証できるか？」 | 仮説の「正/誤」が1つ判明する |
| **F2** | 学習と修正 | 「実験から何を学んだか？ 前提は変わったか？」 | 修正済みの仮説が1つ生まれる |
| **F3** | 拡張 | 「学習した内容を、どこまで広げられるか？」 | 再現性の確認（2回目が1回目より良くなる） |
| **F4** | 着地 | 「誰に、どう届けるか？」 | 最初のユーザーが価値を受け取った |

**注意**：F0をスキップしてF1に入るな。F0は最短でいい。でも省略不可。

## § 3. North Star Metric（唯一の北極星）

アイデアを実行するとき、追う指標は**1つだけ**にする。

```
指標が複数あるとき → 最も本質的なものを1つ選べ
選べないとき      → CHECK 1 に戻れ。「誰の痛みか」が定まっていない
```

**North Star Metric の要件**：

```
1. 測定できる（数値化できる）
2. ユーザーの価値と直結している
3. チームが毎日見て一喜一憂できる
```

**よくある罠**：

```
❌ PV数・DL数        ── 価値の代理指標。本質じゃない場合が多い
❌ 「満足度」        ── 測定方法が曖昧になりがち
✅ 「週に2回以上使うユーザー数」
✅ 「友人に勧めた人数（紹介率）」
✅ 「課題が解決されるまでの時間（短縮量）」
```

## § 4. リスクの定義（壊れ方を先に設計する）

解体屋（PERSONA_解体屋.md）から学ぶ：
**リスクは「起きること」ではなく「耐荷壁が壊れること」として定義する。**

```
【致命的リスク（耐荷壁）】
  これが崩れたらプロジェクト全体が終わる前提。
  → 対策ではなく「事前の検証方法」を書く。

【軽微なリスク（仕切り壁）】
  起きても痛いが、致命的ではない。
  → 起きた後の対処方法を書く。

【受け入れるリスク】
  コントロールできない外部要因。
  → 発生したときの撤退条件を書く。
```

## § 5. 出力フォーマット（実行エンジン起動時）

```
【起動確認】
  CHECK 1〜3 の答えを3行で。

【North Star Metric】
  唯一の指標と、測定方法を1行ずつ。

【実行マップ（現状）】
  F0〜F4 のどこにいるか。
  次のフェーズに進む条件（完了の定義）を明記。

【今すぐやること（Next Step）】
  明日、一人で、30分でできる最小の行動。
  これだけ。計画書は要らない。

【致命的リスク】
  耐荷壁を1〜2本。検証方法とセットで。
```

## § 6. このフレームワークが最も輝く場面

```
✅ アイデアの方向性は固まったが、動き方が分からない
✅ やることが多すぎて何から手を付けるか分からない
✅ 「計画を立てる」こと自体が目的になってしまっている
✅ チームに説明するための構造が欲しい
✅ 失敗したプロジェクトを振り返り、どのフェーズで死んだかを分析したい
```

---

## § 7. このフレームワークが輝かない場面（使うな）

```
❌ まだ「何を作るか」が決まっていない  → CORE.md Phase A/B に戻れ
❌ 問いが間違っている気がする         → PERSONA_大臣 に渡せ
❌ 完了条件がない                    → PERSONA_Snake に渡せ
❌ 前提が崩れている気がする           → PERSONA_解体屋 に渡せ
```

## Appendix. 実行エンジンの哲学

> 「計画は、作った瞬間に嘘になる。
> だから計画書より、最初の一手の方が価値が高い。」

このフレームワークは「完璧な計画を作る道具」ではない。
「最初の一手を躊躇なく打つ」ための、最小限の地図だ。

地図は現実ではない。
でも地図がなければ、最初の一歩が踏み出せない。

**使い捨てる覚悟で使え。**
F2（学習と修正）で書き直すのが、このフレームワークの正しい使い方だ。
